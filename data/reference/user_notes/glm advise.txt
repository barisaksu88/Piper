### Suggestions

#### 1. Add a "Recently Completed" Section

The §13 items in TRIGGER_FLOW.md serve as an implementation log, but the roadmap itself jumps straight from "Queued" to "Backlog." Someone reading only ROADMAP.md has no sense of velocity. Consider a short section at the top:

```markdown
## Recently Completed (retired to TRIGGER_FLOW.md)
- §13.12 Context Arbitration Policy (R-1)
- §13.18 Bulk Mutation Rollback Manifests
- §13.17 State Mutex — Cross-Domain Dependency Check (R-6)
```

This gives visitors immediate confidence that the project is actively shipping.

#### 2. R-5 and R-6 Are Implemented But Still Referenced as Backlog Items

The roadmap mentions "Typed Success Constraints (R-5)" and "State Mutex (R-6)" as implemented (§13.15-13.18), but they don't appear in the Queued or Backlog sections with explicit retirement notes. Consider adding a brief `*Implemented — see §13.15-13.18*` marker similar to what you did for Bulk Mutation Rollback Manifests, so it's clear these are done rather than missing.

#### 3. Computer Use v0 — Consider Adding a "Scope Creep Kill Switch"

Your spec is excellent about non-goals, but I'd suggest one more safety mechanism: a **per-session action budget**. Something like:

```python
MAX_BROWSER_ACTIONS_PER_STAGE = 15  # configurable via config
```

This prevents the Planner from entering an infinite click loop on a dynamic page. The executor already has `_consecutive_fails` tracking, but a budget on *successful* actions would catch the case where the LLM keeps navigating productively but never converges on the goal. When the budget is exhausted, the engine stops and reports progress honestly.

#### 4. Voice ID — Clarify the Inference Strategy

The spec says `pyannote.audio` for speaker embedding, but this is a heavy dependency (~1GB model). Since Piper is Windows-first local, consider:

- **Lightweight alternative**: `speechbrain/spkrec-ecapa-voxceleb` via ONNX export (~30MB) — same accuracy class, much smaller footprint
- **Lazy loading**: Only load the embedding model when voice ID is first requested, not at boot
- **CPU-only path**: Ensure the embedding model works on CPU since GPU may be occupied by the main LLM

The spec's philosophy section (owner-first, everyone is part of the owner's world) is thoughtful and well-articulated — I'd just add a note about the inference resource budget.

#### 5. Pattern Memory — Consider Adding a Decay Mechanism

The current spec says fingerprints appear 3+ times to get promoted. But what if a user's workflow changes? I'd suggest:

- **Time decay**: Patterns not observed in N days get demoted/removed
- **Override signal**: If the user ignores a pattern hint explicitly ("no, I don't want to do that"), decrement the count
- **Seasonality**: The spec mentions "same day of week" as a trigger — consider also time-of-day buckets (morning vs evening patterns differ)

These are small additions that prevent the pattern log from becoming stale noise over time.

#### 6. Missing Backlog Items Worth Considering

Given your architecture, here are a few items that might fit naturally:

| Item | Rationale |
|---|---|
| **Conversation Branching / Fork** | Users often say "actually, let me try a different approach" — currently the Scratchpad is linear |
| **Skill Composition** | Your `core/skills/` directory exists; a spec for composing skills (e.g., "search then summarize to file") would leverage it |
| **Multi-Model Support** | Allow different LLM backends per role (e.g., small model for Router, large for Planner) — your `LlamaServerClient` is already abstracted enough |
| **Exportable Session Reports** | Change journal + Scratchpad could produce a human-readable "what Piper did this session" document |

#### 7. Roadmap Formatting — Add Effort Estimates

Even rough T-shirt sizing (S/M/L) on each item would help contributors know what to pick up. For example:

- Active Image Selection → **S** (pure UI task)
- Voice ID Embedding → **L** (heavy ML dependency + inference pipeline)
- Pattern Memory → **M** (new engine + integration, but self-contained)
- Computer Use v0 → **XL** (already well-broken-down in your implementation order)
