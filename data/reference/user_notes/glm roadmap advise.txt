### The two divergences ⚠️

**1. Pre-LLM bypass check order (§2 vs. code) — Medium priority**

The spec says:
1. Pending search → REPORTER
2. **Document chat heuristic → DOC_FOCUS** ← 2nd
3. Live screen visual → PERSONA
4. Environment query → PERSONA
5. Operational state query → PERSONA

The code actually does:
1. Pending search → REPORTER
2. Proactive trigger → PERSONA *(§13.10 addition)*
3. Route interceptors (UNDO/EXPLAIN/REMINDER_SET)
4. Environment query → PERSONA
5. Operational state query → PERSONA
6. **Document chat heuristic → DOC_FOCUS** ← 6th
7. Live screen visual → PERSONA

The document chat heuristic moved from 2nd to 6th because the §13.10 (proactive monitor) and §13.9/11 (interceptor) bypasses were inserted before it. In practice this is low risk — it's very unlikely a query would match both "document chat" and "environment query" patterns simultaneously. But the spec says `phase_route()` "enforces the first true bypass," implying order is intentional.

**My recommendation:** Update the TRIGGER_FLOW.md to reflect the actual code order. The current order is arguably better — proactive triggers and interceptors should fire before heuristic checks since they're deterministic and cheaper. The spec just needs to catch up.
